# 嵌入式 Linux 的啟動流程

## 🏃‍♂️ 第一棒：ROM Code (IC 的內部引導程式)

* **位置：** 固化在德州儀器（TI）AM335x 晶片內部的唯讀記憶體（ROM）裡。
* **它算什麼：** 這就是你說的 **IC Internal Bootloader**。它是原廠出廠就刻死在晶片裡的，你動不到它。
* **上電瞬間的行為：** 板子一上電，晶片內部的 RAM（內存）只有幾十 KB，而且此時板子上的大記憶體（DDR）還沒被叫醒，根本不能用。ROM Code 醒來後，會依照你按住的 `BOOT` 鍵，決定去讀取 eMMC 還是 **SD 卡（mmc0）**。
* **它的任務：** 去 SD 卡的 FAT32 分區裡找一個特定名字的檔案 —— **`MLO`**，把它讀進晶片內部小小的內部記憶體裡，然後把棒子交給 `MLO`。

---

## 🏃‍♂️ 第二棒：U-Boot SPL (也就是 `MLO` 檔案)

* **位置：** 存放在你的 SD 卡中，檔名必須死死鎖定叫 **`MLO`**。
* **它算什麼：** 它叫做 **U-Boot Secondary Program Loader (次級引導程式)**。
* **為什麼要有它：** 因為 U-Boot 主程式（`u-boot.img`）太大了（好幾百 KB），此時晶片的 DDR 記憶體還沒初始化，根本裝不下。所以原廠設計了一個極小、被裁剪過的先鋒部隊，也就是 `MLO`。
* **它的任務：** **初始化板子上的 512 MB DDR DRAM 記憶體。** DDR 被叫醒後，空間變大了，`MLO` 就會去 SD 卡把主程式 **`u-boot.img`** 搬進 DDR 裡，把棒子交出去。

---

## 🏃‍♂️ 第三棒：U-Boot 主程式 (也就是 `u-boot.img` 檔案)

* **位置：** 存放在你的 SD 卡中，檔名為 **`u-boot.img`**。
* **💡 你的盲點解惑 ⚠️：** **你在串口狂按空白鍵攔截下來的，就是這一棒！** 當你看到 `Hit any key to stop autoboot` 時，你進入的是 `u-boot.img` 的互動命令列（`=>`），**而不是 IC 內部的 ROM Code**。ROM Code 在第一秒跑完就下班了，根本不會理你的鍵盤空白鍵。

### 🎬 U-Boot 拿到棒子後的自動化劇本：

如果你**沒有**按空白鍵，U-Boot 就會開始讀取 SD 卡裡剩下的這三個檔案：

1. **`boot.scr` (開機腳本)：** 這是你用 `mkimage` 打包 `boot.cmd` 生成的二進位檔案。它就像是一份「自動化執行清單」，裡面寫著：*「請幫我把 uImage 讀到記憶體 0x82000000、把 dtb 讀到 0x88000000，然後用 bootm 指令發動開機！」* U-Boot 讀了這份腳本，就會乖乖照做。
2. **`am335x-boneblack.dtb` (設備樹 / 硬體地圖)：**
Linux 核心是個瞎子，它不知道這塊板子上有幾顆 LED、網路孔接在哪個腳位。這個 `.dtb` 檔案就是**給 Linux 核心看的硬體說明書**。U-Boot 會負責把它撈進記憶體，等一下交給核心。
3. **`uImage` (Linux 核心本體)：**
作業系統的靈魂。U-Boot 把它搬進記憶體後，會執行最後一個動作：**把 CPU 的指針指向核心的入口（Entry Point），U-Boot 正式下班。**

---

## 🏃‍♂️ 第四棒：Linux Kernel (也就是 `uImage` 跑起來後)

* **狀態：** 核心在記憶體裡被叫醒，畫面秀出 **`Starting kernel ...`**。
* **它的任務：** 核心透過剛才那份 `dtb` 說明書，開始逐一接管硬體（啟動時鐘、驅動串口、叫醒網卡、初始化 USB）。
* **最後一哩路（rootfs）：**
當硬體都驅動完畢後，Linux 核心需要載入軟體程式（像是登入畫面、你寫的 C 程式）。這時它會去讀取你 SD 卡的第二個大分區 —— **`rootfs` (根檔案系統)**。
它把 `rootfs` 掛載成根目錄 `/`，從裡面讀取 `/bin/sh`，這時你的終端機才會啪一聲，跳出讓你輸入帳號密碼的 `beaglebone login:` 畫面。

---

## 📊 總結：全線檔案分工大表

為了讓你後續開發（BSP 移植、新 IC 流程設計）時思路絕對清晰，請看這份終極分工表：

| 跑第幾棒 | 執行的軟體/檔案 | 存放位置 | 它的核心任務 |
| --- | --- | --- | --- |
| **第一棒** | **ROM Code** | IC 內部 (ROM) | 物理硬體偵測，去 SD 卡找 `MLO`。 |
| **第二棒** | **`MLO`** | SD 卡 (boot 分區) | 叫醒開發板上的 **DDR 記憶體**，並載入 `u-boot.img`。 |
| **第三棒** | **`u-boot.img`** | SD 卡 (boot 分區) | 提供倒數計時（按空白鍵攔截）。負責把 `uImage` 和 `dtb` 搬進 DDR。 |
| **開機劇本** | **`boot.scr`** | SD 卡 (boot 分區) | 告訴 U-Boot 要用什麼參數、把檔案搬到記憶體哪個地址。 |
| **硬體地圖** | **`am335x-boneblack.dtb`** | SD 卡 (boot 分區) | 告訴 Linux 核心這塊板子的針腳與硬體配置（硬體說明書）。 |
| **第四棒** | **`uImage`** | SD 卡 (boot 分區) | Linux 核心本體，負責初始化並全面接管晶片。 |
| **系統本體** | **`rootfs` 分區** | SD 卡 (第二分區) | 存放 `/bin`, `/etc`。提供指令（如 `ls`）與應用程式，開機成功後的日常運作本體。 |

所以，你的 `tasks.json` 之前把所有檔案全部 `cp` 進 `boot` 資料夾是完全對的，因為第三棒的 U-Boot 在開機時，必須在那個 FAT32 的 `boot` 分區裡一次集滿 `boot.scr`、`dtb` 和 `uImage` 這三樣供品，它才能成功把棒子遞交給第四棒的 Linux 核心。
